# 万能导入 V2 — 大模型调用说明

## 1. 概述

本系统在"运单智能导入"场景中集成了大模型，用于**自动分析用户上传的出库单文件（Excel/PDF），智能生成解析规则**，使系统能够从各种非标准格式的出库单中提取运单数据，大幅降低人工配置规则的成本。

### 核心能力

- 自动识别文件结构（标准行表格 / 卡片式 / 矩阵转置）
- 自动映射字段（配送单号、门店、收件人、SKU编码、数量等）
- 识别合并单元格、复合拆分、尾部收件人信息等复杂格式
- AI 分析结果供用户确认，不会自动执行

---

## 2. 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 大模型 | **智谱 GLM-4.5 Air** (`glm-4.5-air`) | 通过 OpenAI 兼容协议调用 |
| API 端点 | `https://open.bigmodel.cn/api/paas/v4` | 智谱 BigModel 平台 |
| AI SDK | **Vercel AI SDK v4** (`ai` 包) | 提供 `generateObject` 结构化输出 |
| Provider | `@ai-sdk/openai` | 基于 OpenAI 兼容协议，不直接使用 Anthropic SDK（版本不兼容） |
| Schema | **Zod** | 定义输出结构，强制执行格式校验 |
| 运行时 | Next.js 16 Edge / Node.js | 服务端调用，无浏览器依赖 |

### 为什么选择 GLM-4.5 Air

- 国内访问速度快，延迟低
- 中文理解能力强，适合物流出库单场景
- OpenAI 兼容 API，无缝集成 Vercel AI SDK
- 成本低，免费额度充足

### 备选模型配置

本系统支持任意 OpenAI 兼容 API，只需修改环境变量：

| 模型 | Base URL | 备注 |
|------|----------|------|
| 智谱 GLM-4.5 Air | `https://open.bigmodel.cn/api/paas/v4` | 当前使用 |
| DeepSeek Chat | `https://api.deepseek.com/v1` | 备选 |
| OpenAI GPT-4o | `https://api.openai.com/v1` | 备选 |

## 4. 触发时机与流程

### 触发条件

用户上传出库单文件后，在界面上**手动点击"AI 智能分析"按钮**。AI 分析不会自动触发，需要用户主动操作。

### 完整交互流程

| 步骤 | 进度 | 操作 | 说明 |
|------|------|------|------|
| 1 | 10% | 正在上传文件... | 前端构建 FormData，将文件 POST 到 `/api/analyze` |
| 2 | 30% | AI 正在分析文件结构... | 服务端读取文件前 50 行，生成文本预览，调用 AI |
| 3 | 80% | 正在生成解析规则... | AI 返回结果，服务端整理为 ParseRule 结构 |
| 4 | 100% | 分析完成 | 前端展示 AI 生成的规则，用户确认或修改 |

### 用户确认环节

AI 分析完成后，前端展示：
- **规则名称**（可编辑）
- **文件类型**（Excel / PDF）
- **解析引擎**（行表格 / 卡片拆分 / 矩阵转置）
- **AI 推测字段列表**（`guessed`，标注为 AI 推测，供用户确认）
- **JSON 配置预览**
- 提供两个操作按钮：
  - **"试解析预览"**：用当前规则试跑，展示解析结果
  - **"保存规则并解析"**：保存规则到数据库并执行正式解析

## 5. Prompt 工程设计

### 5.1 系统角色

```text
你是一位物流出库单解析专家。请分析用户上传的文件样本，生成一个通用的解析规则配置。
```

### 5.2 目标字段定义（10 个字段）

| 字段 | 内部名 | 是否必填 | 说明 |
|------|--------|----------|------|
| 外部编码/配送单号 | `external_code` | 选填 | 用于聚合分组，必须是真正的配送单号 |
| 收货门店 | `store_name` | A组条件必填 | 收货门店/客户名称，非发货仓库 |
| 收件人姓名 | `receiver_name` | B组条件必填 | — |
| 收件人电话 | `receiver_phone` | B组条件必填 | — |
| 收件人地址 | `receiver_address` | B组条件必填 | — |
| SKU编码 | `sku_code` | 必填 | 物品编码 |
| SKU名称 | `sku_name` | 必填 | 物品名称 |
| 数量 | `quantity` | 必填 | 正数 |
| 规格型号 | `spec` | 选填 | — |
| 备注 | `remark` | 选填 | — |

> **收货信息规则**：A 组（`store_name`）与 B 组（`receiver_name` + `receiver_phone` + `receiver_address`）二选一必填。门店单用 A 组，快递单用 B 组。

### 5.3 三种解析引擎

| 引擎 | 适用场景 | 说明 |
|------|----------|------|
| `row` | 标准行表格 / 配送单 | 每行一条运单记录，最常见的格式 |
| `card` | 卡片式记录 / 调拨单 | 一块区域为一条记录，字段分散在区域内 |
| `matrix` | 矩阵转置 / 库存分货表 | 行头为 SKU，列头为门店，需行列转置 |

### 5.4 关键判断规则

1. **`external_code`** 必须是真正的配送单号/订单号，不能是序号、行号
2. **`store_name`** 是收货门店/客户名称，不是发货仓库名（常见错误）
3. **收货人/电话/地址** 在表格外底部区域时，使用 `trailingInfo` 提取
4. **`headerRows`** 表示表头之前的干扰行数（0-based），不包含真正的表头行
5. **`dataEndMarker`** 数据结束标记，如"合计"、"总计"等关键字
6. **PDF 文件**列索引从 0 开始
7. **复合单元格**（如"苹果x5, 香蕉x3"）需标记 `compositeSplit`

### 5.5 输入限制

- 输入样本为文件前 **50 行**的文本预览
- Token 限制：截取前 **4000 字符**
- 超过 4000 字符的样本会被自动截断

## 6. 结构化输出 (Zod Schema)

通过 Vercel AI SDK 的 `generateObject` + Zod Schema，确保 AI 输出严格符合系统要求的 JSON 结构。

### 6.1 顶层结构

```typescript
{
  name: string;           // 规则名称，如"黎明屯配送单-42列"
  fileType: "excel" | "pdf";
  config: ParseConfig;    // 解析配置（见 6.2）
  guessed: GuessedField[]; // AI 推测的字段列表（供用户确认）
}
```

### 6.2 ParseConfig 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `engine` | `"row" \| "card" \| "matrix"` | 解析引擎 |
| `structure.headerRows` | `number` | 表头之前的干扰行数 |
| `structure.titleRow` | `number` | 标题行索引（可选） |
| `structure.dataStartRow` | `number` | 数据起始行索引 |
| `structure.dataEndMarker` | `string` | 数据结束标记关键字（可选） |
| `structure.cardStartMarker` | `string` | 卡片起始标记（可选） |
| `structure.sheetMode` | `"first" \| "all"` | Sheet 模式（仅 Excel） |
| `fieldMappings` | `FieldMapping[]` | 字段映射数组 |
| `card` | `CardConfig` | 卡片引擎配置 |
| `trailingInfo` | `TrailingInfoConfig` | 尾部信息提取配置 |
| `aggregation` | `AggregationConfig` | 聚合配置 |
| `matrixTranspose` | `MatrixTransposeConfig` | 矩阵转置配置 |
| `compositeSplit` | `CompositeSplitConfig` | 复合拆分配置 |
| `compoundGroup` | `CompoundGroupConfig` | 复合分组配置 |

### 6.3 FieldMapping 字段映射

```typescript
{
  target: FieldName;       // 目标字段（如 "sku_code"）
  source: string;          // 源列名或列索引（如 "G" 或 "6"）
  value?: string;          // 固定值（静态字段）
  required?: boolean;      // 是否必填
  transform?: string;      // 转换函数（如 "trim" | "toUpperCase" | "parseInt"）
}
```

### 6.4 GuessedField — AI 推测字段

```typescript
{
  field: FieldName;    // 字段名
  source: string;      // 推测的源位置
  confidence: "high" | "medium" | "low";  // 置信度
}
```

前端会特别标记 `low` 置信度的字段，提醒用户重点确认。

## 7. 调用参数

```typescript
const { object } = await generateObject({
  model,           // glm-4.5-air
  schema,          // Zod Schema（见第 6 节）
  prompt,          // Prompt 文本（见第 5 节）
  temperature: 0.2,  // 低温度，保证输出稳定、确定性高
  maxRetries: 1,     // 失败后重试 1 次
  // maxTokens 使用默认值，未显式限制
});
```

### 为什么 temperature=0.2

- 规则生成需要**确定性**而非创造性
- 低温度减少 AI 幻觉，降低字段映射错误率
- 经过多轮测试，0.2 在准确性和灵活性之间达到最佳平衡

### maxRetries=1

- 第一次失败后自动重试一次
- 总调用次数最多为 2 次（初次 + 1 次重试）
- 重试策略：相同参数重新调用，不修改输入

## 8. 三层降级策略

系统设计了完善的三层降级，确保即使 AI 服务不可用，用户仍能完成导入：

### 第一层：已知文件模板匹配

**`matchKnownFileRule(rows, fileType, fileName)`** — 纯本地处理，不调用 AI

基于文件指纹（列数、表头关键字、文件名）匹配预设模板。目前内置了以下模板：

| 模板 | 识别特征 | 引擎 |
|------|----------|------|
| 黎明屯配送单 | 42 列大表，表头含"配送单号""物品编码" | row |
| 湖南仓发货明细 | 特定列组合 + "发货仓库"关键字 | row |
| 欢乐牧场库存分货表 | 矩阵式布局，行头为 SKU | matrix |
| 多门店分 Sheet 出库单 | 多个 Sheet 各自包含门店数据 | row |
| 门店调拨单-卡片式 | 非标准表格，块状布局 | card |
| 黔寨寨 PDF 配送单 | PDF 格式，特定布局 | row |

### 第二层：AI 大模型分析

**`analyzeFileAndGenerateRule(previewText)`** — 远程调用大模型

当已知模板无法匹配时，调用 AI 进行分析。这是主要路径，支持任意格式的出库单文件。

### 第三层：本地关键字兜底

**`generateLocalRule(rows, fileType)`** — 纯本地处理，不调用 AI

当 AI 服务不可用或超时时，基于表头关键字自动映射字段：
- 扫描前几行，匹配中英文关键字
- 自动检测卡片/矩阵引擎
- 自动检测尾部收件人信息
- 返回 `fallback: true` 标记，前端提示用户"AI 服务暂不可用，已使用本地规则，请人工核对"

### API 路由降级调用顺序

```typescript
let rule;
try {
  rule = matchKnownFileRule(rows, fileType, file.name)
      || await analyzeFileAndGenerateRule(previewText);
} catch (aiErr) {
  console.warn("AI 分析失败，使用本地兜底规则:", aiErr);
  rule = generateLocalRule(rows, fileType);
}
```

## 10. 错误处理机制

### 10.1 AI 调用层面

| 错误类型 | 处理方式 |
|----------|----------|
| 网络超时 | Vercel AI SDK 自动重试 1 次（`maxRetries: 1`） |
| API Key 无效 | 抛出异常 → 降级到本地兜底 |
| Rate Limit | 抛出异常 → 降级到本地兜底 |
| Zod 校验失败 | `generateObject` 自动请求 AI 重新生成 |
| 返回格式异常 | 外层 catch 捕获 → 降级到本地兜底 |

### 10.2 前端层面

| 状态 | 处理方式 |
|------|----------|
| 请求中 | 显示进度条（"AI 正在分析文件结构..."） |
| 正常返回 | 展示 AI 规则结果 |
| `fallback: true` | 展示规则 + 黄色提示："AI 服务暂不可用，已使用本地规则，请人工核对" |
| 网络错误 | 红色错误提示："AI 分析失败，请重试或手动配置规则" |

### 10.3 安全原则

- **AI 生成的规则仅供用户确认**，不会绕过用户直接执行
- 所有 AI 生成字段在 `guessed` 数组中标注置信度
- 低置信度字段前端以 **⚠️ 黄色标记** 显示
- AI 服务不可用时，用户可**手动创建规则**，系统完全可用
- 敏感数据不会发送给 AI（仅发送表头和结构信息，不发送完整数据行）

## 12. SDK 初始化细节

```typescript
// src/lib/ai.ts
async function getAI() {
  // 动态导入，避免 edge runtime 问题
  const [{ generateObject }, { createOpenAI }] = await Promise.all([
    import("ai"),               // Vercel AI SDK
    import("@ai-sdk/openai"),   // OpenAI 兼容 provider
  ]);

  const modelName = process.env.AI_MODEL || "GLM-4.7";
  const apiKey = process.env.OPENAI_API_KEY
              || process.env.ANTHROPIC_AUTH_TOKEN
              || process.env.ANTHROPIC_API_KEY;
  const baseURL = process.env.OPENAI_BASE_URL
               || process.env.ANTHROPIC_BASE_URL
               || "https://open.bigmodel.cn/api/paas/v4";

  const openai = createOpenAI({ apiKey, baseURL });
  return { generateObject, model: openai(modelName) };
}
```

> **为什么用 `@ai-sdk/openai` 而不是 `@ai-sdk/anthropic`**：
> 项目同时安装了两个包，但 Anthropic 包与 Vercel AI SDK v4 存在版本不兼容问题，
> 因此统一使用 OpenAI 兼容协议。智谱、DeepSeek、OpenAI 等均支持此协议。

## 16. 常见问题排查

| 问题 | 可能原因 | 解决方法 |
|------|----------|----------|
| AI 分析返回本地兜底规则 | API Key 无效或余额不足 | 检查 `.env.local` 中的 `OPENAI_API_KEY` |
| 字段映射不准确 | 表头关键字不常见 | 手动调整，或添加已知模板 |
| 分析超时 | 网络问题或模型繁忙 | 检查 `OPENAI_BASE_URL` 是否可访问 |
| `Cannot find package '@ai-sdk/openai'` | 依赖未安装 | 执行 `pnpm install` |
| Edge Runtime 报错 | 构建配置问题 | 检查 `next.config.ts` 中的 `serverExternalPackages` |
